# 魔法肉鸽卡牌游戏插件开发规范文档 (AGENTS.md)

本文档面向在此项目上进行开发和维护的 AI Agent 及开发人员，定义了本项目的目录结构、代码开发规范以及测试准则。

## 1. 项目目录结构

整个插件的目录结构设计如下：

- `main.py`: 插件主要入口，用于注册 /rogue 命令，进行多级子指令分发并截获未处理异常。
- `metadata.yaml`: 插件元数据配置文件，声明插件版本、唯一标识等。
- `AGENTS.md`: 本项目规范文档（本文档）。
- `game/`: 游戏核心逻辑包。
  - `__init__.py`: 包初始化空文件。
  - `models.py`: 基础状态数据结构模型（使用 Python dataclasses 结构实现）。
  - `cards.py`: 静态卡牌配置数据库（兼容桥接导出 `ALL_CARDS` 与 `MINION_SKILLS` 配置）。
  - `card_impl.py`: 基础卡牌类与各卡牌的具体数据驱动及多态子类实现，并注册 `ALL_CARDS`。
  - `minion_impl.py`: 随从技能模板、各随从特有技能子类实现，并注册 `ALL_MINIONS` 与生成 `MINION_SKILLS`。
  - `amulet_impl.py`: 护符模板、各护符特有钩子回调实现，并注册 `ALL_AMULETS`。
  - `enemy_impl.py`: 怪物模板、Boss（远古红龙）等特殊怪物的意图与行为多态实现类，并注册 `ALL_ENEMIES`。
  - `event_impl.py`: 事件及选项模板、各特殊事件选项逻辑子类实现，并注册 `ALL_EVENTS`。
  - `effects.py`: 游戏基础原子效果类，用于数据驱动组装。
  - `manager.py`: 本地 JSON 存档的读取、保存与销毁管理逻辑。
  - `renderer.py`: 将当前的游戏状态渲染为兼容 OneBot 的纯文本。
  - `engine.py`: 回合制战斗状态推进、出牌动作点消耗、随从控制指令及节点转换控制器。
- `data/`: 存档保存路径（运行时自动创建），包含 `user_{user_id}.json` 存档。
- `scratch/`: 本地临时测试与仿真目录。
  - `test_game.py`: 包含交互式与自动随机仿真的控制台测试程序。

## 2. 代码规范

为确保多端运行顺畅与规范统一，在编写代码时必须严格遵守以下规范：

### 2.1 注释约束 (重要)
- 除非有用户明确的直接指示，否则绝对不要在任何 Python 源代码（包括 main.py, game/*.py, scratch/*.py）中编写任何以 # 开头的单行注释或三引号多行注释。
- 绝对不要删除项目原本已存在的既有注释（如 main.py 中 helloworld 函数的原有注释及 docstring），必须将其原样保留。

### 2.2 视觉与渲染规范
- 游戏中的渲染逻辑全部集中于 game/renderer.py。
- 游戏渲染给机器人的交互文本输出中，禁止使用任何 Markdown 标记（例如加粗 **, 斜体 *, 表格符 |, 代码块包围 ```）及 LaTeX 公式，防止在 QQ OneBot 平台无法正常显示。
- 游玩过程中可以使用 Emoji 和 Unicode 符号进行界面美化与排版对齐。

### 2.3 核心数据与状态规范
- 数据结构须继承 models.py 中的 Dataclass，状态转换与更新均须通过 manager.py 中的 SaveManager完成序列化存盘。
- 在执行消耗资源的行为（如打出卡牌、随从攻击）时，在消耗相应动作点（A/BA/AA）前必须对其合法性进行验证，验证通过后再进行状态的原子修改。
- 玩家随从或护符卡牌打出前，必须预先调用 _get_free_grid(p) 检查玩家战场是否有空余格子（上限 5 个），若满格则进行安全拦截。
- 敌方已移除随从系统，所有敌人均为独立的 EnemyState，放置在 enemies 列表中，数量无上限，且召唤敌人也将在 enemies 列表中新增格子存放。我方随从攻击需消耗随从独立的攻击动作点（AA）。
- 所有的卡牌、随从技能、护符、怪物行为及地图事件选择逻辑均应采用数据驱动或多态继承设计。添加、修改此类资源时，应当在对应的 `_impl.py` 中编写子类继承，并实现具体的 `execute`、`on_end_turn`/`on_take_damage`、`execute_intent` 等方法，并注册入相应的全局映射中，避免在战斗或地图核心引擎中编写硬编码的分支判断。
- 所有的玩家属性偏好设置（如 `fold_guide` 状态指南折叠）需持久化记录于 `PlayerState` 中，并在 `manager.py` 的序列化层进行平滑兼容读取。
- 游戏内包含通用的 Buff 系统。任何对玩家、随从或怪物的属性增益/减益能力牌，必须实现为向目标实体的 `buffs` 列表添加 `BuffState` 对象。在添加时须先检查同名 Buff 是否已存在，若存在则累加 `stacks` 计数实现完美堆叠。
- 战斗引擎对卡牌多次触发（如“回响形态”Buff）应在 `play_card` 中通过判断当前回合已打出卡牌计数器 `cards_played_this_turn` 触发 `stacks` 次的额外调用来实现，并在回合开始/结束时将该计数器清零。
- 卡牌在被打出时会判断 `card.exhaust` 属性，若为 `True` 则自动将其送入 `exhaust_pile`（消耗牌堆）进行归档，否则移入 `discard_pile`。
- 当有随从战死或敌方怪物被击败时，必须统一将其标识录入玩家的 `graveyard`（墓地）列表中，为未来的复活/唤亡能力预留基础。
- 游戏内包含卡牌稀有度系统，卡牌分为普通（common）、稀有（rare）和珍奇（epic）品质。稀有度属性必须在 `card_impl.py` 后方的稀有度映射字典中统一配置，并在开启新游戏时，利用品质权重算法随机调配出“5张普通、2张稀有和1张珍奇”的初始卡组。
- 重构为 AOE 范围伤害的法术卡牌（如火球术、雷鸣波）在执行伤害结算时，应直接在 `execute` 内对 `run.enemies` 中的所有敌方状态进行遍历，并最后通过统一的生命值归零过滤出死亡敌人放入墓地并 pop 移出战阵，严禁直接使用会因为 pop 索引产生下标漂移的旧版单体 `_damage_target` 逻辑。

### 2.4 指令调度与队列系统
- 指令解析与执行分支（包括 `使用`、`随从`、`选择`、`特殊`、`结束`、`折叠` 等操作）均由 `main.py` 的统一分发器 `_execute_sub_action` 路由。为保证队列系统的兼容性与高鲁棒性，所有指令执行时须以元组 `(res_text, should_terminate)` 形式返回。
- 队列系统 (`队列` / `q` / `queue`) 须支持递归嵌套执行，并通过 `split_by_comma_with_brackets` 括号匹配深度算法正确隔离各层指令里的逗号，且在队列内部必须原生兼容所有的常规单条操作文本格式。如果在任意步骤检测到终结状态（如阵亡或战局胜利），后续队列必须立即发生截断。

## 3. 测试规范

本项目的验证应当严格按照自动化测试和边界测试进行：

### 3.1 本地测试手段
- 本地交互式控制台测试：在插件根目录下执行：
  py scratch/test_game.py
  可在终端交互使用指令模拟游玩，验证指令容错性。
- 本地自动化仿真：在插件根目录下执行：
  py scratch/test_game.py --auto
  仿真程序将执行 150 步的随机卡牌和路线动作，涵盖战斗、商店购买、事件处理、篝火休息、战利品奖励。

### 3.2 提测与通过指标
- 自动随机测试在 150 步内没有发生未捕获异常退出（即程序返回 exit code 0 且无 Python Traceback），方可判定为核心引擎通过测试。
- 在修改 models.py 字段时，需确保 manager.py 中的 to_dict 和 from_dict 序列化层同步更新，且不破坏存量存档文件的兼容性。
