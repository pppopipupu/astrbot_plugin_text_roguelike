# 魔法肉鸽卡牌游戏插件开发规范文档 (AGENTS.md)

本文档面向在此项目上进行开发和维护的 AI Agent 及开发人员，定义了本项目的目录结构、代码开发规范以及测试准则。

## 1. 项目目录结构

整个插件的目录结构设计如下：

- `main.py`: 插件主要入口，用于注册 /rogue 命令，进行多级子指令分发并截获未处理异常。
- `metadata.yaml`: 插件元数据配置文件，声明插件版本、唯一标识等。
- `AGENTS.md`: 本项目规范文档（本文档）。
- `game/`: 游戏核心逻辑包。
  - `__init__.py`: 包初始化空文件。
  - `models.py`: 基础状态数据结构模型（使用 Python dataclasses 结构实现）。
  - `cards.py`: 静态卡牌配置数据库（兼容桥接导出 `ALL_CARDS` 与 `MINION_SKILLS` 配置）。
  - `card_impl.py`: 基础卡牌类与各卡牌的具体数据驱动及多态子类实现，并注册 `ALL_CARDS`。
  - `minion_impl.py`: 随从技能模板、各随从特有技能子类实现，并注册 `ALL_MINIONS` 与生成 `MINION_SKILLS`。
  - `amulet_impl.py`: 护符模板、各护符特有钩子回调实现，并注册 `ALL_AMULETS`。
  - `enemy_impl.py`: 怪物模板、Boss（远古红龙）等特殊怪物的意图与行为多态实现类，并注册 `ALL_ENEMIES`。
  - `event_impl.py`: 事件及选项模板、各特殊事件选项逻辑子类实现，并注册 `ALL_EVENTS`。
  - `effects.py`: 游戏基础原子效果类，用于 data-driven 组装。
  - `manager.py`: 局内 JSON 存档的读取、保存与销毁管理逻辑。
  - `renderer.py`: 将当前的游戏状态渲染为兼容 OneBot 的纯文本。
  - `battle_engine.py`: 回合制战斗状态推进、出牌动作点消耗、随从控制指令及节点转换控制器。
- `scratch/`: 本地临时测试与仿真目录。
  - `test_game.py`: 包含交互式与自动随机仿真的控制台测试程序。

## 2. 代码规范

为确保多端运行顺畅与规范统一，在编写代码时必须严格遵守以下规范：

### 2.1 注释约束 (重要)
- 除非有用户明确的直接指示，否则绝对不要在任何 Python 源代码（包括 main.py, game/*.py, scratch/*.py）中编写任何以 # 开头的单行注释或三引号多行注释。
- 绝对不要删除项目原本已存在的既有注释（如 main.py 中 helloworld 函数的原有注释及 docstring），必须将其原样保留。

### 2.2 视觉与渲染规范
- 游戏中的渲染逻辑全部集中于 game/renderer.py。
- 游戏渲染给机器人的交互文本输出中，禁止使用任何 Markdown 标记（例如加粗 **, 斜体 *, 表格符 |, 代码块包围 ```）及 LaTeX 公式，防止在 QQ OneBot 平台无法正常显示。
- 游玩过程中可以使用 Emoji 和 Unicode 符号进行界面美化与排版对齐。
- 敌方在场上的意图展示被整合拼接为 (A: xx, BA: xx) 的分立格式。

### 2.3 核心数据与状态规范
- 数据结构须继承 models.py 中的 Dataclass，状态转换与更新均须通过 manager.py 中的 SaveManager 完成序列化存盘。
- 玩家每回合开始重置及战斗初始化时的基础动作扩容为 2A 1BA。
- 引入 0 动作消耗牌机制，当卡牌的 cost_a 与 cost_ba 为 0 时打出不扣除动作资源。精简 2A 卡牌配置以维系连招体验，仅有火球术和雷鸣波两张全屏大招为 2A 消耗。
- 随从动作成长机制重构：在回合交替时，随从的 actions 和 bonus_actions 实行 += 自增累加（无限增长），不再被直接重置。随从的 AA (attack_actions) 攻击动作点则保持每回合开始时重置为 1 的规则。
- 随从品质动作与技能消耗限制：只有 epic 品质的随从 (奥术傀儡 arcane_golem) 每回合重置并累加 BA；其它所有非 epic 品质随从每回合仅有并累加 1A (无 BA)。同时，非 epic 随从的技能消耗统一提升为 2A 或者是 3A 消耗以匹配无限动作积攒设计，仅有 epic 随从保留了 2BA 消耗技能。
- 敌人多行动机制重构：为所有敌人引入 A 和 BA 的动作点分配。普通怪与爪牙为 1A 0BA，精英为 1A 1BA，BOSS 远古红龙为 1A 2BA。重置时按照其各自的最大上限进行还原。敌人在其回合中，会依次消耗动作点并执行其对应的 A 意图、BA 意图与 BA2 意图，执行前实时做生存判定和截断。
- 爪牙与胜利判定机制：魔仆与狂暴猎犬作为召唤出来的爪牙敌人 (is_summon == True)，当当前场上存活的所有敌人全部是爪牙时，系统将直接判定玩家胜利。
- 规范化持久化存储：存档系统摒弃了本地硬编码 data/ 相对路径。在 MyPlugin 初始化时，必须通过 StarTools.get_data_dir("astrbot_plugin_text_roguelike") API 获取机器人官方统一指定的插件专属隔离数据目录进行读取和写入。同时使用 try-except 防御性降级机制，若在没有 AstrBot 框架的环境下运行（如本地 test_game.py 测试），则平滑回落至本地 data 目录下，防范仿真测试崩溃。

### 2.4 指令调度与队列系统
- 指令解析与执行分支（包括 使用、随从、选择、特殊、结束、折叠 等操作）均由 main.py 的统一分发器 _execute_sub_action 路由。为保证队列系统的兼容性与高鲁棒性，所有指令执行时须以元组 (res_text, should_terminate) 形式返回。
- 批量指挥随从：随从命令的格子参数支持 all、所有、* 或逗号分隔，引擎会对匹配的我方随从串行执行同一动作，并在中途如遇胜负或玩家阵亡安全自动截断。
- 随从队列简写：支持类似卡牌队列的逗号分隔随从动作队列，可自动被预解析模块转换、路由到队列引擎中顺序执行。
- 回合结束输出优化：在输入 结束 (或 e) 回合后，系统会在新一轮玩家回合开始时，自动在新回合提示文本行下方追加渲染输出最新状态的战场画面与新抽取的手牌 GameRenderer.render_game(run)，极大化操作流畅性。

## 3. 测试规范

本项目的验证应当严格按照自动化测试和边界测试进行：

### 3.1 本地测试手段
- 本地交互式控制台测试：在插件根目录下执行：
  py scratch/test_game.py
  可在终端交互使用指令模拟游玩，验证指令容错性。
- 本地自动化仿真：在插件根目录下执行：
  py scratch/test_game.py --auto
  仿真程序将执行 150 步的随机卡牌和路线动作，涵盖战斗、商店购买、事件处理、篝火休息、战利品奖励。

### 3.2 提测与通过指标
- 自动随机测试在 150 步内没有发生未捕获异常退出（即程序返回 exit code 0 且无 Python Traceback），方可判定为核心引擎通过测试。
- 在修改 models.py 字段时，需确保 manager.py 中的 to_dict 和 from_dict 序列化层同步更新，且不破坏存量存档文件的兼容性。
