# 魔法肉鸽卡牌游戏插件开发规范文档 (AGENTS.md)

本文档面向在此项目上进行开发和维护的 AI Agent 及开发人员，定义了本项目的目录结构、代码开发规范以及测试准则。

## 1. 项目目录结构

整个插件的目录结构设计如下：

- `main.py`: 插件主要入口，用于注册 /rogue 命令，进行多级子指令分发并截获未处理异常。
- `metadata.yaml`: 插件元数据配置文件，声明插件版本、唯一标识等。
- `AGENTS.md`: 本项目规范文档（本文档）。
- `game/`: 游戏核心逻辑包。
  - `__init__.py`: 包初始化空文件。
  - `data/`: 静态数据配置包。
    - `__init__.py`: 包初始化空文件。
    - `card_data.py`: 全体卡牌的属性、伤害/护盾/治疗参数与反馈字符串模板（含传奇卡牌）。
    - `minion_data.py`: 我方全体随从的属性、技能参数与反馈模板。
    - `enemy_data.py`: 敌方全体角色意图、召唤物属性与意图说明数据。
    - `amulet_data.py`: 全体护符的默认耐久与结算效果参数。
    - `buff_data.py`: 全体战斗 Buff 的展示名称与效果描述配置。
    - `event_data.py`: 荒野事件的场景叙述、选项参数及选项触发反馈模板（支持多阶段配置）。
    - `relic_data.py`: 全体遗物的属性、稀有度、售价与效果描述配置。
  - `models.py`: 基础状态数据结构模型（使用 Python dataclasses 结构实现）。
  - `cards.py`: 静态卡牌配置数据库（兼容桥接导出 `ALL_CARDS` 与 `MINION_SKILLS` 配置）。
  - `card_impl.py`: 基础卡牌类与各卡牌的具体数据驱动及多态子类实现，并注册 `ALL_CARDS`。
  - `minion_impl.py`: 随从技能模板、各随从特有技能子类实现，并注册 `ALL_MINIONS` 与生成 `MINION_SKILLS`。
  - `amulet_impl.py`: 护符模板、各护符特有钩子回调实现，并注册 `ALL_AMULETS`。
  - `relic_impl.py`: 遗物辅助解析、展示名与描述获取等函数。
  - `enemy_impl.py`: 怪物模板、Boss（远古红龙）等特殊怪物的意图与行为多态实现类，并注册 `ALL_ENEMIES`。
  - `event_impl.py`: 事件及选项模板、各特殊事件选项逻辑子类实现，并注册 `ALL_EVENTS`。
  - `effects.py`: 游戏基础原子效果类，用于 data-driven 组装。
  - `manager.py`: 局内 JSON 存档的读取、保存与销毁管理逻辑。
  - `renderer/`: 将当前的游戏状态渲染为兼容 OneBot 的纯文本的模块包。
    - `__init__.py`: 导出统一的 GameRenderer 类，代理至各子模块。
    - `menu.py`: 菜单、卡牌库及玩家牌组渲染。
    - `battle.py`: 战斗阶段及详细战斗快照信息渲染。
    - `map.py`: 地图路由选择预览及先古节点渲染。
    - `explore.py`: 随机事件、奇妙商店、篝火营地、战利品与古老宝箱渲染。
    - `query.py`: 随从、遗物、Buff及卡牌模糊匹配查询效果渲染。
  - `battle_engine.py`: 回合制战斗状态推进、出牌动作点消耗、随从控制指令及节点转换控制器。
  - `map_engine.py`: 关卡地图网络拓扑结构生成、移动路由控制及特殊关卡初始化逻辑。
- `scratch/`: 本地临时测试与仿真目录。
  - `test_game.py`: 包含交互式与自动随机仿真的控制台测试程序。
  - `test_flow.py`: 单元测试流程，用于验证第一层先古契约与商店购买、地图生成。

## 2. 代码规范

为确保多端运行顺畅与规范统一，在编写代码时必须严格遵守以下规范：

### 2.1 注释约束 (重要)
- 除非有用户明确的直接指示，否则绝对不要在任何 Python 源代码（包括 main.py, game/*.py, scratch/*.py）中编写任何以 # 开头的单行注释或三引号多行注释。
- 绝对不要删除项目原本已存在的既有注释（如 main.py 中 helloworld 函数的原有注释及 docstring），必须将其原样保留。

### 2.2 视觉与渲染规范
- 游戏中的渲染逻辑全部集中于 game/renderer/ 模块包中。
- 游戏渲染给机器人的交互文本输出中，禁止使用任何 Markdown 标记（例如加粗 **, 斜体 *, 表格符 |, 代码块包围 ```）及 LaTeX 公式，防止在 QQ OneBot 平台无法正常显示。
- 游玩过程中可以使用 Emoji 和 Unicode 符号进行界面美化与排版对齐。
- 敌方在场上的意图展示被整合拼接为 (A: xx, BA: xx) 的分立格式。
- 引入树形地图列表预览机制，列表层级渲染玩家当前所处层及往后推 3 层的全部路径及可达节点去向，规避非等宽字体排版错乱。
- 玩家在进入战斗、事件、商店等界面时，顶部如果拥有遗物则会自动拼装显示为 `🎒 遗物: 【xxx】` 列表。

### 2.3 核心数据与状态规范
- 数据结构须继承 models.py 中的 Dataclass，状态更新均须通过 manager.py 中的 SaveManager 完成序列化存盘，`map_data` 存入 `GameRun` 且 `relics` 存入 `PlayerState`。
- 玩家初始手牌上限由 8 提升为 12。若拥有遗物【虚空面具】，则手牌上限将被强行限制为 9。
- 玩家每回合开始重置及战斗初始化时的基础动作扩容为 2A 1BA。
- 引入 0 动作消耗牌机制，当卡牌的 cost_a 与 cost_ba 为 0 时打出不扣除动作资源。
- 随从动作成长机制：随从在回合交替时 actions 和 bonus_actions += 累加，AA 每回合重置为 1。非 epic 随从无 BA，且技能提高为 2A 或 3A。
- 敌人行动与爪牙：普通/爪牙 1A 0BA，精英 1A 1BA，BOSS 远古红龙 1A 2BA，当存活敌人全为爪牙时自动判定玩家获胜。
- 引入眩晕（Stun）状态：处于该状态下的敌人在下一回合无法行动，并扣除一层眩晕 buff 恢复。
- 护盾流失机制：在每回合开始时（即各自回合开始时），玩家和所有存活敌人的护盾（护甲）均会自动失去一半（向下取整），并在回合更替的反馈信息中提供流失详情提示。
- 玩家手牌回合结束弃牌与保留机制：在玩家回合结束时，手牌中不具备保留（retain）效果的所有卡牌将被自动丢弃并移入弃牌堆。具有保留效果的卡牌继续保存在玩家手牌中。回合结束的丢弃操作绝不会触发卡牌的灵巧（agile）效果。

### 2.4 地图与契约机制
- 地图系统全长为 20 层，第 1 层固定为先古契约节点，第 5 和 15 层必定为古老宝箱房，第 10 和 20 层必定为首领战（第 10 层中首领，第 20 层最终首领），第 11 层为先古节点。
- 地图在进入第 2 层和第 12 层时分别一次性生成前半程或后半程的树形随机拓扑路径图并序列化存储。
- 开局先古契约提供 3 个随机选择项：界面石碑数量修改为三座，从“先古遗物”池和“先古卡牌”池中分别随机抽取 1 或 2 个选项（两者数量相加为 3 且随机选取，保证开局 3 个选项中至少包含 1 个先古遗物和 1 个先古卡牌）。
- 第 11 层先古赐福依然是纯正面奖励，传奇卡牌池中加入流星爆与大法师的祈愿这两张传奇卡牌，并同样保持随机生成 3 个不重复的赐福包。
- 战斗与地图节点规范化：当在地图中选择 elite 或 boss 类型的节点去往战斗时，应当在 map_engine.py 内部统一将 run.node_type 规范化为 battle，以避免渲染阶段发生“未知关卡状态”的错误。
- 古老宝箱房要求玩家选择卡组中一张牌移除作为献祭，随后开启获得金币、随机珍奇卡牌与随机遗物奖励。

### 2.5 遗物（Relic）系统与商店联动
- 遗物分为普通、稀有、珍奇三种稀有度，永久生效。
- 遗物与卡牌的具体效果配置如下：
  - 【先古之眼】首回合多抽 2 卡；【准备背包】首回合多抽 1 卡且 +1BA；【重装甲片】/【坚固皮革】初始护盾 +8/+4；【铁锈之锁】/【贪婪契约】开局扣血 -4/-3；【能量核心】每回合 +1A；【红龙之血】胜利回血 5；【磨刀石】随从攻击 +1；【奥术符文】/【狂暴之印】法术伤害 +1/+2；【枯萎之种】禁止战斗治疗。
  - unstable_crystal (不稳定水晶): 玩家使用的法术伤害增加 1 点，且每回合开始时额外获得 1BA；但每当玩家使用法术牌时，玩家受到 1 点伤害。
  - vampiric_touch (吸血之触): 玩家打出伤害法术时回复 1 点生命值，但最大生命值上限永久减少 4 点。
  - ancient_page (先古残页): 战斗开始时，额外将 2 张【奥术星火】加入手牌，但战斗开始时玩家失去 4 点生命值。
  - blind_spot (盲目之障): 每场战斗第一回合开始时，玩家少抽 2 张牌。
  - tax_contract (税收契约): 每当进入下一个关卡时，玩家失去 6 金币。
  - meteor_swarm (流星爆): 消耗 2A 0BA，对所有敌人造成 8d12 随机火焰伤害。消耗。
  - archmage_wish (大法师的祈愿): 消耗 1A 1BA，获得 10 点护盾，本场战斗法术伤害 +4（祈愿奥术），并抽 2 张牌。消耗。
- 奇妙商店中会随机刷新出 1 个未拥有的遗物商品；若拥有【黄金罗盘】所有商品享受 6 折，拥有【贪婪契约】所有商品额外享受 8 折，两个折扣乘算叠加。

### 2.6 指令调度与队列系统
- 指令分支由 main.py 的统一分发器 _execute_sub_action 路由。所有指令以元组 (res_text, should_terminate) 形式返回。
- 随从指令支持 comma 逗号分隔的队列预解析，支持 all、* 等批量指挥随从指令。
- 回合结束指令会自动追加渲染战场最新画面和新抽取的手牌。
- 核心指令支持中英文变体解析：例如“开启”支持英文别名“start”。
- 提供“帮助”及“help”专用子命令，调用 GameRenderer.render_help() 列出全量指令的使用教程和中英文别名，当输入未知指令时系统应当引导用户使用帮助指令。
- 随从攻击指令的自动目标选择：当发起随从攻击指令但未指定具体目标格子时，系统会自动将第一个存活的敌方单位作为攻击目标，保持与攻击型卡牌一致的行为。

### 2.7 数据驱动与配置分离规范
- 静态配置数据化：所有卡牌、随从、怪物、护符、遗物、Buff、事件一律禁止硬编码，必须存放在 `game/data/` 的独立数据字典中。
- 反馈模板化：各实现类利用 `.format()` 动态渲染 `feedback` 模板并进行回填。

## 3. 测试规范

本项目的验证应当严格按照自动化测试和边界测试进行：

### 3.1 本地测试手段
- 本地交互式控制台测试：在插件根目录下执行：
  py scratch/test_game.py
  可在终端交互使用指令模拟游玩，验证指令容错性。
- 本地自动化仿真：在插件根目录下执行：
  py scratch/test_game.py --auto
  仿真程序会自动识别“进行中的游戏”提示并发送“开启 确认”强制覆盖重启，打印实时的命令交互日志，随机执行 150 步卡牌释放和行路动作。
- 本地单元流验证：在插件根目录下执行：
  py scratch/test_flow.py
  模拟并检验开局契约选卡、地图展示、遗物在商店刷出等主要步骤。
- 本地卡牌保留与回合丢弃验证：在插件根目录下执行：
  py scratch/test_retain.py
  验证回合结束时自动丢弃手牌、卡牌保留和灵巧卡牌绕过机制的正确性。

### 3.2 提测与通过指标
- 自动随机测试在 150 步内没有发生未捕获异常退出（即程序返回 exit code 0 且无 Python Traceback），方可判定为核心引擎通过测试。
- 修改 dataclass 时确保 manager.py 中的 to_dict 和 from_dict 序列化层同步更新，且向下兼容老版本存档。
