## 1. 专有架构设计与代码规范

在编写或扩展对决模式代码时，必须严格遵守以下模式专有架构与开发规范：

### 1.1 对决指令与动作路由解耦规范 (命令模式)
- 局外指令与局内动作应彻底解耦，禁止在 DuelRouter 中编写臃肿的大型 if-else 或 switch 逻辑分支。
- 任何新增的局外指令必须继承自 DuelCommandHandler（定义于 game/core/duel/base.py）并在 game/core/duel/commands.py 中实现。
- 任何新增的局内动作必须继承自 DuelActionHandler（定义于 game/core/duel/base.py）并在 game/core/duel/actions.py 中实现。
- 基类中提供 __init_subclass__ 机制，在子类加载时会自动通过 names 参数自动发布和装配到全局路由注册表，实现高扩展性和开闭原则支持。

### 1.2 逻辑状态结算委派规范 (单一职责)
- DuelEngine 不应无限膨胀为上帝类。所有涉及到随从攻击、随从技能判定与消耗、幸运币判定以及卡牌强化的详细状态变更和校验，必须统一由 DuelResolver（定义于 game/core/duel/resolver.py）具体负责实现。
- DuelEngine 仅保留最小的门面方法作为对外一致 API 兼容。

### 1.3 状态同步与渲染规范
- 任何局内动作的存档存盘、两端（公开战局与两端私密手牌/详情消息）的渲染、以及提取追加 log 消息，必须统一通过 DuelRouter._save_and_render_state 方法处理，杜绝多处复制粘贴渲染代码。
- 任何动作执行后的领主 HP 归零检测及 GP 结算逻辑，必须统一委派至 DuelRouter._check_victory` 方法完成。

### 1.4 血量与无职业属性
- 对决模式中，双方玩家初始生命值上限与当前生命值均统一为 200 点。无任何子职业系统。

### 1.5 TCG 卡牌流转与能量机制
- 玩家回合开始时仅抽 1 张卡牌，且每回合结束时**不丢弃手牌**。
- 能量成长：第一回合上限为 1A 1BA，后续每 2 个自己的回合增长 1A 能量上限，每 3 个自己的回合增长 1BA 能量上限，最大上限为 10 点，每回合开始时能量重置为当前上限。
- 后手方在第一回合开始前自动获得 2 个“幸运币”（非卡牌形态），随时可在己方回合打出以增加 1A 动作点。
- **随从与护符特殊消耗机制**：随从与护符卡牌被打出时，判定为特殊的消耗状态并触发被消耗事件（CardExhaustEvent），卡牌 ID 不会进入消耗堆（exhaust_pile），也不会进入弃牌堆（discard_pile），而是直接从牌组大循环中彻底离场。普通的消耗性卡牌被打出时，依然会正常移入消耗堆并触发消耗事件。

### 1.6 召唤失调与攻击重构
- 普通随从召唤进场的回合无法进行攻击（通过将其攻击动作点 attack_actions 初始化扣减为 0 来限制，无 summon_sickness 实体 Buff）。
- **突进 (Rush)**：具有突进词条的随从进场当前回合可发起攻击，但目标只能指定为敌方随从（e2~e7），无法攻击敌方领主（e1）。
- **冲锋 (Charge)**：具有冲锋词条的随从进场当前回合可自由选择攻击领主（e1）或随从。
- **召唤行动点重置**：在下一回合开始（即随从经历回合交替）后，在场我方随从的突进限制解除，攻击动作点重置为 1，可自由选择攻击敌方随从或领主。
- 随从攻击反击：我方随从攻击敌方随从时，若目标随从在受到伤害后存活，它会对攻击我方的随从发起等同于其攻击力的攻击力反击。

### 1.7 伤害卡牌打脸限制
- 对决模式中绝大部分法术及攻击卡限制为“只能选择随从为目标”，只有配置声明 face_target=True 的直伤卡牌才被允许指定敌方领主为目标。

### 1.8 进化点机制
- 双方开局各自拥有 4 个进化点。自第 3 回合开始解锁进化点使用。
- 每回合仅限进化一次。支持进化手牌、我方随从或我方护符为 + 强化版。随从进化时生命补满且上限+2，攻击+2；护符进化时吟唱时间（countdown）保持不减。

### 1.9 局外牌组与校验规则
- 开始对决前必须拥有当前活动的合规牌组。
- 牌组卡牌总数必须在 25~50 张之间，且同名基础卡牌的携带量上限为 4 张。

### 1.10 双端渲染与多端显示适配
- 局内发起对战时，群聊中只显示公开的战局简报（血量、能量、随从及护符状态等）。手牌、幸运币和详细指引通过 QQ 私信（DM）下发给双方。
- 在 Matrix 等不解析纯换行符的第三方聊天平台中，如果消息体是含有 ━━━━━━━━━━━━━━━━━━━━ 分界线的对决公开战局或私密详情，系统会自动检测并整体包裹在 ``` Markdown 代码块中，以确保正常换行和等宽字体排版。

## 3. 局外菜单与动作队列规范

### 3.1 对决局外主菜单与状态指令
- 局外唤起：当处于对决局外（无激活的对局），且输入空子指令（如仅输入对决）或者输入状态、s、status指令时，系统必须返回包含对决模式玩法操作说明和玩家当前活动牌组详情（包含牌组卡牌张数、牌组合法性状态校验结果）的对决局外主菜单。
- 局内状态：在战斗进行中，若玩家输入状态、s、status指令，则触发战局快照的重新推送，重新向群聊发送公开战局，并私信发送该玩家本人的手牌详情等私密信息。

### 3.2 对决队列动作执行机制
- 队列指令：局内支持使用队列、q、queue作为前缀或者仅使用逗号分割来顺序执行动作队列（如：队列 [使用 1 e1, 使用 2]）。
- 中括号剥离：动作队列所接收的指令参数支持使用成对的中括号包裹，系统在分割执行前会自动识别并将其剥除。
- 缩写指令智能扩展：在顺序循环执行队列动作时，若某一步动作的第一个词为纯数字，系统必须在交由具体处理器执行前将其自动补齐并转换为标准的打牌动作（如 1 转换为 使用 1），以确保多端输入的便捷性。
- 分割隔离性：对决动作队列使用的切割函数必须是定义在对决基础库中的字符切割解析器，严禁使用肉鸽模式的跨包依赖。

## 4. 对决模式开发与扩展指引

### 4.1 对决模式卡牌开发与扩展指引
- 数据配置分离：新增对决卡牌时，需在 game/data/duel_card_data.py 中的 DUEL_CARD_CONFIG 字典里添加其静态属性定义（包含 cost_a、cost_ba、rarity、type、desc等）。
- 卡牌效果注册：需在 game/entities/cards/duel.py 中实现具体的卡牌效果逻辑类，继承自 DuelCardImpl / CardImpl，并使用装饰器 @register_card(cid) 进行全局注册。
- 0 注释原则：在新增卡牌类时，严禁编写任何 # 井号注释和 docstring 说明文档。

### 4.2 播报及反馈文案修改指引
- 系统消息数据驱动：对决模式中所有的局内外系统广播、错误提示、操作反馈、胜负判决和帮助文本，均统一定义在 game/data/duel_template_data.py 中的 DUEL_BROADCAST_TEMPLATES 字典中。
- 禁止在逻辑代码中硬编码任何中文提示语。所有提示文案的修改必须只在 duel_template_data.py 中进行，并在调用处通过 DUEL_BROADCAST_TEMPLATES["key_name"].format(...) 动态注入。

### 4.3 指令与动作扩展指引
- 局外指令：若要为对决模式新增局外指令，必须在 game/core/duel/commands.py 中编写继承自 DuelCommandHandler 类的具体处理器类，并通过 names 参数传入其支持 of 指令名别名列表。
- 局内动作：若要新增局内打牌动作，必须在 game/core/duel/actions.py 中编写继承自 DuelActionHandler 类的具体动作处理器，并通过 names 参数传入支持的动作名别名列表。
- 装配机制：系统在运行初始化加载 commands.py 和 actions.py 时，会自动利用 __init_subclass__ 钩子将指令类和动作类实例化并挂载到全局路由中，无需手动在路由器中编写额外的 switch 分支。
