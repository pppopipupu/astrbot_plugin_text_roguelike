## 1. 专有代码规范

在编写肉鸽模式相关代码时，必须严格遵守以下模式专有规范：

### 1.1 视觉与渲染规范
- 敌方在场上的意图展示被整合拼接为 (A: xx, BA: xx) 的分立格式。
- 引入树形地图列表预览机制，列表层级渲染玩家当前所处层及往后推几层的全部路径及可达节点去向，规避非等宽字体排版错乱。
- 玩家在进入战斗、事件等界面时，顶部如果拥有遗物则会自动拼装显示为带有遗物图标的文本列表。
- 除非明确提及，否则游戏文案及战斗日志输出中，严禁插入任何形式的解释性括号（例如“（特殊消耗状态）”等）。
- 卡牌词条（如“消耗。”、“保留。”、“灵巧。”、“易碎 X。”等）必须统一置于卡牌描述的最后，不得放在开头或介绍文本的中间。

### 1.2 核心数据与状态规范
- **卡牌ID及修饰彻底解耦与 CardState 强类型规范**：卡牌大循环堆（deck, hand, draw_pile, discard_pile, exhaust_pile, graveyard）中的卡牌彻底解耦为非 str 继承的 @dataclass CardState。所有卡牌状态变更（如升级、镶嵌宝石、重放、易碎等）只修改 CardState 对象的属性。
- **from_cid 弃用及卡牌 ID 禁改规范**：在新写的代码中绝对禁止使用 from_cid（已标记为弃用方法，仅在加载旧存档或测试用例兼容时由底层库静默使用），且绝对禁止更改卡牌 id 或为卡牌 id 拼接任何后缀（如升级不能修改 ID，不能为卡牌 ID 追加 + 或 :gems 等后缀，ID 必须保持其原始注册名不变，通过 upgraded 布尔值等属性标识状态）。
- 玩家初始手牌上限为 12。若拥有特定限制上限的被动或遗物，手牌上限会被限制为相应数值（如 9）。每回合开始时玩家默认抽取 6 张卡牌。特别地，在战斗第一回合开始前，会先将整套大牌组洗牌存入抽牌堆。
- 玩家每回合开始重置及战斗初始化时的基础动作扩容为 2A 1BA。
- 随从动作机制：在玩家回合交替时，我方随从的 actions 增加 1（即 actions 累加），随从的攻击动作点（AA）每回合重置为 1。随从不具备 BA系统，所有随从的技能消耗均只使用 A 动作点。
- 敌人行动与爪牙：普通/爪牙与精英角色拥有各自的动作分配规则，首领 BOSS 在每回合行动点通常会获得对应的专属额外动作扩容。当存活敌人全为爪牙时自动判定玩家获胜。
- 引入眩晕（Stun）状态：处于该状态下的敌人在下一回合无法行动。其意图重写、动作点归零完全由 StunBuff 中的 on_enemy_sync_intents 监听 EnemySyncIntentsEvent 处理并扣除眩晕 Buff，严禁在控制器的核心逻辑中硬编码对 "stun" 字符串的特异判定。
- 回响形态触发机制：每回合打出的卡牌额外打出，每张牌最多回响 10 次，多余的层数会顺延回响每回合后续打出的卡牌。
- 护盾流流失机制：在各自回合开始时，玩家和所有存活敌人的护盾均会自动失去一半（向下取整），并在回合更替的反馈信息中提供流失详情提示。
- 玩家手牌回合结束弃牌与保留机制：在玩家回合结束时，手牌中不具备保留（retain）效果的所有卡牌将被自动丢弃并移入弃牌堆。具有保留效果的卡牌继续保存在玩家手牌中。回合结束的丢弃操作绝不会触发卡牌的灵巧（agile）效果。
- 玩家战场随从格数量机制：玩家战场的随从和护符格子数量上限固定为 6 个，编号为 1 至 6，目标指向代码依次对应为 p1 至 p6。
- 卡组选择与移除序号映射机制：卡组列表展示时进行去重和字母升序排序渲染。在进行卡牌献祭或净化服务等需要通过序号移除卡组卡牌的场景时，系统内部将输入序号映射到经过排序去重后的卡组卡牌种类上，并对该种类的卡牌移除一张。
- 伤害类型系统与护盾穿透机制：引入了完整的 14 种伤害类型系统（挥砍 slashing、穿刺 piercing、钝击 bludgeoning、火焰 fire、寒冷 cold、闪电 lightning、雷鸣 thunder、强酸 acid、毒素 poison、黯蚀 necrotic、光耀 radiant、心灵 psychic、力场 force、真实 true），真实（true）类型伤害能够无视并完全穿透护盾。允许在宏观的卡牌描述、Buff 说明及遗物规则中继续使用“物理伤害”（代表挥砍、穿刺、钝击的合称）或“魔法伤害”作为宏观叙述，但在任何具体的伤害结算、伤害播报及战斗日志输出中，绝对禁止出现“造成了 X 点物理/魔法伤害”的描述，必须明确输出具体的伤害类型名称（例如，随从和怪物普通物理攻击必须直接使用并显示为“钝击伤害”，狂暴撕咬等意图归为“穿刺伤害”）。
- 伤害播报格式：伤害播报统一为逗号分隔的扁平格式，即 对【xxx】造成 xx 点 xx 伤害，对护盾造成 xx 伤害，对生命造成 xx 伤害（仅当数值大于 0 时输出对应项），不使用括号，且如果生命/护盾某一个变动数值为 0 则不予播报该项；若变动数值均为 0（即未造成实际伤害），则播报尾部自动变更为 （但xxx免疫了这次攻击！） 的形式。
- 护符谢幕曲与入墓机制：护符吟唱时间归零销毁时，卡牌ID自动移入我方随从墓地（minion_graveyard），并触发独特的谢幕曲（Last Words）效果（如获得金币并过牌、获得护盾、造成单体或全体真实伤害、回复生命、全体伤害等）。
- 随从与护符特殊消耗机制：随从与护符卡牌被打出时，被判定为特殊的消耗状态并触发被消耗事件（CardExhaustEvent），其卡牌ID不会进入消耗堆（exhaust_pile），也不会进入弃牌堆，从牌组大循环中彻底离场（直到护符吟唱结束入墓或被随从消耗逻辑处理）。
- 随从进场效果规范化：随从卡牌被打出召唤时触发的效果在播报中统一称作“入场曲”。
- 战斗专注卡牌机制：打出战斗专注卡牌时为 0A 0BA 免费牌，立即抽取指定数量的卡牌，并给玩家附加名为无法抽牌（tactical_focus）的临时 Buff。在拥有该 Buff 期间，玩家无法通过任何卡牌、遗物或随从效果再行抽牌，该 Buff 每回合结束时减少 1 层并随层数归零清除（层数代表剩余回合数）。
- X费卡牌机制：卡牌的 cost_a 或 cost_ba 设为 -1 时，代表打出时会消耗当前玩家所拥有的全部 A 或 BA（即扣减为 0），其扣减量会被记录在 run.node_data 的 last_x_cost_a / last_x_cost_ba 属性中作为 X 计数。X 费卡牌 must 在所有展示界面中显式渲染为“(消耗: X A)”、“(消耗: X BA)”或“(消耗: X A Y BA)”等格式。禁止在卡牌的 desc 中硬编码任何如“消耗所有的动作”等与消耗本身逻辑重复的文案描述。
- 职业主属性规范：战士职业初始生命值上限为 80，开局自带 23 张红色物理卡牌与中立卡牌库；法师职业初始生命值上限为 45。战士不具备子职业。
- 战士专属被动「动作如潮」：每次战斗开始前自动初始化限用次数为两次。在战斗中可通过技能指令 /rogue 技能 as、/rogue sk 或 /rogue k 直接触发（无第二个参数默认为 "as" 补全），每次行动能够额外获得 2A 动作点。
- 物理伤害与力量多倍机制：重写力量 Buff 计算，使其仅在打出物理类型伤害（slashing 挥砍、bludgeoning 钝击、piercing 穿刺）的卡牌时生效，并且力量对于重刃（heavy_blade）造成的伤害加成在结算时自动乘以 3（未升级）或 5（升级）。
- 物理卡牌双发克隆机制：双发卡牌会挂载 double_tap_buff。在打出下一张物理伤害牌时，复制并就地重放该卡牌效果，重放次数完成后该 Buff 自动移除。
- 坚不可摧（Unyielding）机制：为特定实体（如 24 层的腐化之心首领）提供的一回合内最多承受 X 点非真实伤害的限制状态。超出的部分会直接免疫，但真实伤害可以穿透此限制且不计入累计额度。在该实体所属的回合结束时，已受伤害累计额会被自动重置。
- 亚弗戈蒙（Aforgomon）Boss 独特战斗机制：在第 31 层首领战中，亚弗戈蒙拥有一阶段 180 HP（2A 1BA，释放时间之链、门扉内爆、时空扭曲）和强力二阶段觉醒变身拦截机制。濒死时，其名字变更为“【时空主宰】亚弗戈蒙”，恢复至 220 HP 并挂载专属的“时空主宰”与“钟摆共振”机制：【时空主宰】被动使其每回合开始获得 20 护盾、净化所有负面 Buff、受伤反弹 3 点真实伤害并有 50% 几率从时间裂缝中召唤“时空残影”分担 50% 伤害；【钟摆共振】惩罚玩家回合结束动作点全光的行为（受打出卡牌数 × 2 的真实伤害），若保留了至少 1A 1BA 则奖赏下回合额外多抽 2 张卡牌；二阶段强力技能【时序断裂】会将手牌中除首张外的卡牌全部磨损附着“易碎 1”。

### 1.3 地图与契约机制
- 地图系统包含多层关卡推进。某些固定层级包含初始契约节点、宝箱房、首领战或特殊先古赐福节点。
- 地图在进入特定路段前，会一次性生成前半程或后半程的树形随机拓扑路径图并序列化存储。
- 开局契约提供 3 个随机选择项，从遗物池和卡牌池中分别随机抽取，保证开局包含一定数量的遗物或卡牌，支持玩家进行初始奖励的选取。
- 先古赐福节点是纯正面奖励，在奖励卡牌池中支持随机生成 3 个不重复的赐福包。
- 战斗与地图节点规范化：当在地图中选择 elite 或 boss 类型的节点去往战斗时，应当在 map_engine.py 内部统一将 run.node_type 规范化为 battle，以避免渲染阶段发生“未知关卡状态”的错误。
- 宝箱房要求玩家选择卡组中一张牌移除作为献祭，随后开启获得一定战利品奖励。
- 所有随机卡牌获取的场景统一修改为提供三张备选卡牌以供三选一，且均支持跳过获取。

### 1.4 稀有度系统与联动
- 全体卡牌与遗物有且仅有以下 7 个稀有度（对应英文和中文映射如下）：
  - 普通 (common)
  - 稀有 (rare)
  - 极品 (epic)
  - 传奇 (legendary)
  - 神话 (mythic)
  - 神器 (artifact)
  - 诅咒 (curse)：用于标识所有的纯负面被动（如各种纯负面的遗物或卡牌）。
- 遗物效果永久生效。
- 商店商品支持多项特定折减价格的被动或遗物效果，所有的商品折扣计算将采用乘算叠加规则。

### 1.5 指令调度与队列系统
- 随从指令支持 comma 逗号分隔的队列预解析，支持 all、* 等批量指挥随从指令。
- 回合结束指令会自动追加渲染战场最新画面和新抽取的手牌。
- 核心指令支持全量中英文变体解析：其中“牌组”新增英文别名 deck，“总览”新增别名 overview，“放弃”新增别名 abandon（同时放弃确认支持 confirm 参数）。
- 引入用户独立的“免前缀肉鸽模式 (Rogue Mode)”：可以使用 mode 或 模式 指令控制开关。此状态持久化于用户的生涯数据中，默认对所有用户关闭。当某个用户开启后，其发送的无前缀消息若以有效游戏指令开头，或首个词为数字且该用户拥有活跃存档，则会被直接拦截匹配作为游戏指令执行，保障普通群聊闲聊消息放行不受误拦截。
- 随从攻击指令的自动目标选择：当发起随从攻击指令但未指定具体目标格子时，系统会自动将第一个存活的敌方单位作为攻击目标，保持与攻击型卡牌一致的行为。
- 局外购买子指令别名支持：支持使用 buy 别名执行购买动作。
- 职业选择指令别名与快捷索引支持：在职业（class）管理中，支持使用 c 代替 选择，且选择时支持直接指定子职业序号。

## 2. 测试规范

本项目的肉鸽模式验证应当严格按照自动化测试和边界测试进行：

### 2.1 本地测试手段
- 本地控制台交互式游玩：在项目根目录下执行：
  py play_rogue.py
  即可直接免前缀在命令行中游玩游戏。
- 本地交互式控制台测试：在插件根目录下执行：
  py scratch/test_game.py
  可在终端交互使用指令模拟游玩，验证指令容错性。
- 本地自动化仿真：在插件根目录下执行：
  py scratch/test_game.py --auto
  仿真程序会自动识别“进行中的游戏”提示并发送“开启 确认”强制覆盖重启，打印实时的命令交互日志，随机执行 150 步卡牌释放和行路动作。
- 本地综合单元测试：在插件根目录下执行：
  py scratch/test_flow.py
  使用标准 unittest 框架，依次测试游戏核心业务及边界场景。

### 2.2 提测与通过指标
- 自动随机测试在 150 步内没有发生未捕获异常退出（即程序返回 exit code 0 且无 Python Traceback），方可判定为核心引擎通过测试。

### 2.3 单元测试编写规范
- 源码零注释约束：所有的测试代码与测试脚本均视为源码的一部分，其中绝对禁止包含任何以井号字符开头的注释 and docstring 文档字符串。
- 免前缀与用户输入模拟：新编写的单元和集成测试禁止直接调用底层 cli_router.handle_command，必须默认开启 stats.rogue_mode = True 以启用免前缀模式，并且全部通过调用 run_command(plugin, "command_string") 传入完整命令文本来直接模拟真实用户的输入行为。
- 目标血量保护设计：在测试伤害类卡牌（如冰霜新星、深渊崩塌等）时，为防止测试用敌人在前几轮打牌中直接死亡并被系统从 enemies 列表中删除，导致后续测试指令执行因找不到目标而报错，应当在测试点中将敌方目标的 hp 和 max_hp 设置为高数值（如 9999）。
